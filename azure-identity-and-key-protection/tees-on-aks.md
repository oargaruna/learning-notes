# TEEs Available on AKS

Four TEE options, three different granularities.

## Process-level: Intel SGX

- **VM SKUs**: `DCsv2`, `DCsv3`, `DCdsv3` (Coffee Lake / Ice Lake with SGX).
- **Mechanism**: per-process enclaves with CPU-encrypted memory (EPC). Small EPC budget (up to ~256 GB on Ice Lake, historically 128–192 MB per process before paging).
- **AKS integration**: `confcom` addon installs the SGX device plugin and DCAP attestation components. Pods request `sgx.intel.com/epc` as a resource.
- **SDKs**: Open Enclave, EGo (Go), Gramine / Occlum (lift-and-shift unmodified Linux binaries).
- **Attestation**: DCAP → Intel PCS → MAA (or your own verifier).
- **Use when**: you want the smallest TCB (just enclave code + SGX libs), or you need per-process isolation even within one VM. Painful to port existing apps; syscalls are expensive.

## VM-level: Confidential VM node pools

Whole node is a CVM. Guest OS, kernel, and everything in it run in encrypted memory. Host / hypervisor / fabric cannot read node RAM.

- **AMD SEV-SNP**: `DCasv5`, `DCadsv5`, `ECasv5`, `ECadsv5`. GA.
- **Intel TDX**: `DCesv5`, `DCedsv5`, `ECesv5`, `ECedsv5`. GA.
- **Attestation**: vTPM-based boot attestation + CPU-signed hardware report, normalized by MAA as `x-ms-attestation-type: sevsnpvm` or `tdxvm`.
- **Use when**: lift-and-shift existing workloads without code changes, and you trust all pods co-resident on the node. Binding granularity is the node, so SKR release policies only pin to node-level measurements — weaker for multi-tenant clusters.

## Pod-level: Confidential Containers (CoCo)

Built on SEV-SNP CVM node pools, using Kata Containers to give each pod its own lightweight CVM.

- **Runtime class**: `kata-cc-isolation`.
- **Underlying SKUs**: same SEV-SNP `DCasv5` / `ECasv5` family. TDX-backed CoCo is on the roadmap; SEV-SNP is the GA path.
- **Mechanism**: each pod boots into a minimal utility VM. The Kata agent inside measures OCI image layers, command, env, and mounts, pushing the hash into `x-ms-sevsnpvm-hostdata`. A security policy (generated by `genpolicy` from the pod YAML) constrains what the agent will accept at runtime — blocks `kubectl exec`, arbitrary mounts, etc., unless policy allows them.
- **Binding granularity**: specific image digest + pod spec. This is the shape that makes per-workload SKR policies meaningful.
- **Use when**: multi-tenant clusters, untrusted admins, or SKR tied to a specific workload rather than a node.

## Accelerator-level: Confidential GPU (preview / limited regions)

- **NVIDIA H100 in confidential computing mode** paired with SEV-SNP CVM node pools (`NCCadsH100v5`).
- Memory between CPU and GPU is encrypted over PCIe; GPU produces its own attestation report that chains with the CPU's.
- Intended for confidential AI inference / training where weights or input data are sensitive.
- MAA supports compound attestation (CPU + GPU) so SKR policies can require both.

## Picking between them

| Need | Pick |
|---|---|
| Minimal TCB, willing to rewrite | SGX |
| Lift-and-shift, single-tenant node | SEV-SNP or TDX CVM node pool |
| Per-pod isolation, multi-tenant, SKR pinned to workload | Confidential Containers (SEV-SNP + Kata) |
| Confidential AI on GPUs | H100 CVM node pool |

For the SKR scenario where the key should be released only to a specific workload, with CSI/kubelet cut out of the path, **Confidential Containers** is the right match — the `hostdata` claim binds to the pod image, not the node.
